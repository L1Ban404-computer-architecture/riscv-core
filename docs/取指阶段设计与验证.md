# 取指阶段设计与验证

IF 负责顺序 PC 分配、CoreBus 请求保持、请求/响应配对、redirect 后旧响应丢弃，
以及 IF/ID 返回队列。

## 参数与结构

| 参数 | 默认值 | 作用 |
| --- | ---: | --- |
| `FetchOutstandingDepth` | 1 | 已接受但尚未消费响应的最大请求数 |
| `IfIdQueueDepth` | 2 | 已返回、等待 ID 的指令数 |

```text
PC -> request holding register -> CoreBus
                 request fire -> PC FIFO
CoreBus response + PC FIFO -> fetch FIFO -> ID
```

holding register 保证请求受背压时 valid 和地址稳定。PC FIFO 使用 fall-through
模式，允许请求与响应同周期完成；fetch FIFO 关闭满载同拍替换，切断 ID ready
到 CoreBus response ready 的组合路径。

## Redirect 与旧响应

redirect 同周期完成：

- IF/ID valid 组合屏蔽；
- fetch FIFO 清空；
- PC 更新为目标；
- 尚未暴露的 holding request 清除。

已经暴露或完成握手的 CoreBus 请求不能撤销。由于响应严格有序，IF 使用
`discard_count_q` 表示队首需要丢弃的旧路径响应数。重复 redirect 重新以当前
outstanding 数量覆盖丢弃范围；受背压的已暴露请求在随后接受时也计入旧路径。
因此不存在有限宽度 generation/epoch 回绕。

## 异常

指令响应 error 转换为 instruction access fault，`tval` 为请求 PC。异常与原始
响应数据组成正常 IF/ID 事务，后级只使用异常语义。

## 验证

```sh
make test ALL=if_stage
make test ALL=if_stage WAVE=fst
```

定向和随机测试覆盖复位启动、请求保持、零延迟响应、IF/ID 背压、参数化深度、
访问错误、redirect 时请求等待、旧响应丢弃，以及旧响应返回前连续两次 redirect。
