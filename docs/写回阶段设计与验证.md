# 写回阶段设计与验证

本文档说明 `wb_stage` 的唯一架构提交点、写回、trap/MRET、退休 trace 语义，以及
对应的模块级验证。异常与 CSR 的完整契约见 `docs/异常与CSR设计与验证.md`。

## 设计目标

WB 是五级顺序流水线的退休点，负责：

- 把 MEM/WB 中的最终写回事务送回 ID 内部寄存器堆。
- 内部拥有 `csr_unit`，为 EX 提供组合 CSR 读端口。
- 按固定优先级提交 trap entry、MRET 和普通 CSR 写。
- 对 trap/MRET 产生架构 redirect 和后端 `flush_backend`。
- 在事务真正退休时产生 `core_debug_o.valid`。
- 将随指令移动的扁平 debug payload 形成最终退休 trace。
- 保证 store、branch 等不写 GPR 的指令仍能正常退休。

## 组合退休端

MEM/WB `stream_register` 已经是流水线的最后一道时序边界。当前核心没有
外部退休背压，因此 WB 不需要再放置寄存器或 FIFO：

```text
mem_wb_ready_o = 1
wb_fire        = mem_wb_valid_i && mem_wb_ready_o
```

`wb_stage` 自身不再增加退休流水寄存器，但内部拥有时序 CSR 状态，因此需要
`clk_i/rst_ni`。MEM/WB payload 仍由 MEM 内部的 `stream_register` 保持，CSR 只在
`wb_fire` 派生的提交使能下更新。

## 写回语义

`wb_fire=1` 且本条指令不是 trap/MRET 时，WB 透传 `mem_wb_bus_i.wb_req`。
`wb_fire=0` 或异常覆盖普通副作用时，`wb_req_o` 清零，防止残留 payload 误写。

`wb_req_bus_t` 保留两层有效语义：

- `valid`：该指令存在 GPR 写回意图。
- `data_valid`：`wdata` 已经是最终可写入值。

寄存器堆的实际写使能为：

```text
wb_req.valid && wb_req.data_valid && rd_addr != x0
```

x0 过滤由 `regfile` 统一负责，WB 不改写原始退休事务的 `rd_addr`。

合法 CSR 指令可以在同一个提交沿同时写 CSR 和把旧值写入 GPR。若本条事务带异常，
则 trap entry 覆盖 CSR/GPR 写；MRET 同样不产生普通写回。

## Trap 与 MRET 提交

提交优先级为：

```text
trap entry > MRET > 普通 CSR 写；合法 CSR 的 GPR 旧值写回可同拍发生
```

trap 使用故障指令 PC 写 `mepc`，并重定向到提交前 `mtvec`。MRET 重定向到提交前
`mepc`；若目标非四字节对齐，则 MRET 转化为 cause 0 异常，不执行返回状态转换。
两者均产生 `flush_backend`，清除全部年轻后端事务。

## Debug 退休 trace

`core_debug_o.valid` 仅在 `wb_fire` 时置位。其余字段只保留退休观察需要的
架构事件：

```text
pc / instr
gpr_we / gpr_waddr / gpr_wdata
mem_op / mem_size / mem_addr / mem_data
redirect_valid / redirect_target_pc
trap / intr / cause / tval
mstatus / mtvec / mepc / mcause / mtval
```

五个 CSR 快照在 RTL 内以 `csr_state_bus_t` 传递，只有最外层 `riscv_core` 为兼容
runner ABI 将其展平为独立端口。

`gpr_we` 由 `wb_req.valid && wb_req.data_valid` 折叠得到。`mem_sign_ext`、
原始 `mem_rsp.rdata`、译码 `ctrl` 和 EX 中间 `alu_result` 不进入最终 trace；
`mem_op` 使用 NONE/READ/WRITE 编码；成功 load 的访问数据同时进入 `mem_data`，
其中只有访问宽度覆盖的低位字节有效。load 的符号/零扩展结果也体现在 `gpr_wdata`。

异常指令仍退休一条 trace，但 `gpr_we=0`、`mem_op=NONE`。五个 CSR 字段是当前
提交后的快照。bubble 周期除稳定快照外不产生退休事件；Debug 不参与功能控制或
前递判断。

## 时序关系

MEM/WB payload 在一个周期内同时驱动：

- `wb_req_o`，由 ID 中的 regfile 在周期末写入。
- `core_debug_o`，供仿真环境在该退休周期采样。

EX 不从 WB 输出取前递值，而是直接观察同一条 MEM/WB payload，避免一条内容
重复的顶层回路。

## 验证

模块级验证位于 `tests/wb_stage/`：

| 测试 | 覆盖内容 |
| --- | --- |
| `valid_transaction_writes_back_and_flattens_debug` | 有效写回和所有退休 debug 字段展平。 |
| `instruction_without_register_write_still_retires` | store 类无 GPR 写回指令仍产生退休事件。 |
| `bubble_masks_stale_payload` | 无效 MEM/WB payload 不产生写回或退休，x0 请求保持原语义。 |

运行测试：

```sh
make test ALL=wb_stage
```

生成波形：

```sh
make test ALL=wb_stage WAVE=fst
make test ALL=wb_stage WAVE=vcd
```
