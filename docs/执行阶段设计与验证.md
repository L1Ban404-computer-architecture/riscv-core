# 执行阶段设计与验证

EX 对 ID/EX 事务执行操作数选择、前递、ALU、控制流、地址检查和 CSR 组合运算，
并通过单入口 `stream_register` 输出 EX/MEM。所有副作用仍只在后级提交；EX 只形成
请求和提交元数据。

## 数据通路

```text
ID/EX -> source-use decode -> forwarding -> operand select -> ALU
                                      |                 |
                                      |                 +-> branch target
                                      +-> store data        memory address
                                                        CSR new value
                                                               |
                                                               v
                                                          EX/MEM register
```

前递优先级为：

```text
EX/MEM > MEM pending {valid, rd} > MEM/WB > ID/EX captured value
```

EX 只比较指令真正使用的源寄存器。pending load 命中时停顿；若 MEM/WB 的短暂前递值
在停顿期间出现，`forwarding_unit` 将其保存到当前事务执行，避免退回 ID 锁存的旧值。

## 控制流与异常

分支、JAL 和 JALR 在事务真正进入 EX/MEM 时才产生 redirect。JALR 先清除目标 bit 0，
随后按 IALIGN=32 检查；目标等于顺序 PC+4 时不产生多余 flush。`redirect_bus_t`
只携带 valid 和目标，具体指令类别由退休指令本身表达。

EX 在没有上游异常时按优先级补充：

- 已采用控制流目标未对齐；
- load/store 地址未自然对齐；
- CSR 组合读端口报告未实现地址。

出现异常后关闭 redirect、访存和 GPR/CSR 写请求。CSR/SYSTEM/异常事务进入 EX 前
等待所有更老 EX/MEM、LSU outstanding 和 MEM/WB 事务排空，因此 CSR 旧值无需专用
前递。

## 输出契约

EX/MEM 包含功能 PC、访存请求、写回候选、异常、提交控制和完整退休 debug。load 的写回候选在
EX 中只有目标寄存器、数据尚未有效；store 保留原始 rs2 数据，由 MEM 根据地址低位
生成 lane 对齐的 wdata/wstrb。

EX/MEM 输出在 `valid && !ready` 时保持稳定；backend flush 同周期屏蔽输入并清除
寄存器，防止年轻事务产生请求。

## 验证

`tests/ex_stage/test_ex_stage.py` 覆盖 ALU/操作数选择、分支与跳转、前递优先级、
load-use 停顿、停顿期间前递值保存、访存请求和弹性寄存器背压。CSR、异常和 flush
的跨级行为由整核测试覆盖。

```sh
make test ALL=ex_stage
```
